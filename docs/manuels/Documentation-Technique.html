1.3 Flux de données

Architecture request-response standard (web) Navigateur → Next.js App Router → Supabase (PostgreSQL + Auth).
Les API Routes /api/* utilisent la service role key pour les opérations admin (contournement RLS).

Pour chaque module, le patron de données est :

  1. Hook TanStack Query (usePayments, useStudents…) — requête Supabase, mise en cache, invalidation ciblée
  2. Composant — consomme le hook, affiche les données, déclenche mutations
  3. Mutation — appelle directement Supabase ou une API Route, puis queryClient.invalidateQueries(KEY)
  4. Contextes — AuthContext (session + rôle), NotificationContext (toasts), PaymentConfigContext (frais)

2. Authentification & Sécurité

2.1 Supabase Auth (SSR)

L'authentification utilise @supabase/ssr (v0.10.3) avec des cookies HTTP-only pour la persistance de session. Trois clients Supabase distincts coexistent :

ClientFichierClé utiliséeUsage
Browserlib/supabase/client.tsANON_KEYComposants client (lecture, mutations RLS)
Serverlib/supabase/server.tsANON_KEY + cookiesServer Components, middleware
Adminlib/supabase/admin.tsSERVICE_ROLE_KEYAPI Routes admin (bypass RLS)

Flux login admin/agent

  1. Saisie username ou email + mot de passe
  2. Lookup dans les comptes hardcodés (fallback local dev)
  3. supabase.auth.signInWithPassword(email, password)
  4. Récupération profil table users → stockage localStorage.currentUser
  5. Si must_change_password = true → ChangePasswordModal bloquant
  6. Redirection selon rôle : /etudiant (student) | /tableau-de-bord (staff)

Flux login étudiant

  1. Saisie username (ex : jean.dupont)
  2. Construction email Auth : jean.dupont@students.joda.app (fonction buildStudentAuthEmail)
  3. supabase.auth.signInWithPassword(authEmail, password)

Forgot password

  • Route POST /api/forgot-password — génère mot de passe temporaire Joda@XXXXX, persiste, envoie email/SMS
  • Rate-limit anti-DoS : 1 réinitialisation max par user toutes les 5 min (in-memory, retourne 200 silencieusement)
  • Les étudiants ont un contact_email réel dans la colonne users.contact_email (différent du @students.joda.app Auth)

2.2 Middleware proxy.ts

Convention Next.js 16 Le fichier s'appelle proxy.ts (non middleware.ts) — convention interne du projet.
  • Appelle getUser() (validation serveur + refresh automatique du token)
  • Routes publiques : /login, /api/forgot-password, /auth/callback, /api/auth/exchange
  • Toute autre route → redirection /login si non authentifié

2.3 Row Level Security (RLS)

Migration FIX_RLS_RECURSION_v5.sql — à exécuter dans Supabase Dashboard (en attente).

  • Fonction JWT get_auth_user_role() lit auth.jwt() -> 'app_metadata' — zéro requête DB, zéro récursion
  • Toutes les politiques RLS référencent cette fonction au lieu de joindre la table users
  • Les API Routes admin contournent le RLS via le client supabase-admin (service role key)

8. Notifications Email & SMS

8.1 Email (Resend)

Fichier : src/app/lib/emailService.ts

L'application utilise Resend (v6.12.2) pour l'envoi d'emails transactionnels. Resend fournit une API SMTP moderne et fiable pour les emails d'application.

FonctionDéclencheur
sendPaymentReminderCron check-late-payments (retard détecté)
sendPaymentResultEmailValidation ou rejet d'un paiement (admin)
sendPaymentDeclarationEmailDéclaration de paiement par l'étudiant
sendDocumentSubmissionEmailSoumission de documents par l'étudiant
sendStudentMessageEmailMessage envoyé via le module Communication
sendWelcomeEmailCréation d'un compte étudiant ou utilisateur

Templates HTML responsive — design rouge/blanc, bilingues FR/EN selon getLang(langue).

8.2 SMS (smsvas.com)

Fichier : src/app/lib/smsService.ts

  • Fonction principale : sendSmsToPhone(phone, message)
  • Normalisation automatique numéros camerounais → 237XXXXXXXXX
  • 160 caractères max (SMS simple)
  • Envoyé en parallèle de l'email sur tous les événements clés

8.3 Cron — check-late-payments

  • Route : GET /api/cron/check-late-payments (protégée par x-api-key: CRON_SECRET)
  • Fréquence recommandée : quotidien (00h00 ou 06h00)
  • Actions : détecte paiements en retard → statut retard → email + SMS rappel
  • Délai de grâce + pénalité par jour configurables dans PaymentConfigContext
